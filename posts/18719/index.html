程序化交易广告

《计算广告》第六章笔记

  • 广告主希望在每次展示上独立出价，要求询价、出价、竞价在展示时进行，产生了以实时竞价（RTB）为核心的程序化交易市场

  • 广告交易平台（ADX）：用 RTB 的方式实时得到广告候选，并依照竞价进行投放

  • 与 ADX 对应的采买方为 DSP，供给方为 SSP

    • DSP 需要具备点击率预估、点击价值估计、流量预测、站外推荐等多方面的计算能力
    • SSP 需要在众多变现手段中，动态选择变现价值最高者，最大化媒体收益

  • 程序化交易催生了数据加工和交易市场，两个关键产品为数据交易平台（data exchange）和数据管理平台（DMP）

    • 数据交易平台：从第三方数据入手，提供数据源服务
    • 数据管理平台：从第一方数据入手，提供数据加工服务

  • 程序化交易的出现，使得在线广告市场向着数据驱动、计算导向的方向发展

1 实时竞价

  • 广告网络中既有的人群标签无法完成广告主所有的投放需求，这些需求需要广告主自有的数据才能实现，即定制化用户标签（customized audiance segmatation）
  • 用定制化用户标签指导广告投放，是实时竞价的关键产品目标
  • 实时竞价的交易方式不仅解放了效果类广告需求，也为品牌广告创造了全新的机会

1.1 实时竞价的流程

  1. 用户标识映射（cookie 映射）

    1. 从广告主网站向 DSP 服务器发起 cookie 映射请求
    2. DSP 与 ADX 服务器之间通信，完成 cookie 映射

  2. 广告请求（ad call）

    1. 用户接触到媒体网站的广告位

    2. 媒体网站的前端向 ADX 发起广告请求

    3. ADX 向各 DSP 发送 URL（或应用 ID）和用户标识，发起询价请求。随后，DSP 根据数据决定是否参与竞价，如果参与则计算并返回自己的出价。在等待一个固定的时间片后，ADX 选出出价最高的 DSP，返回给媒体网站

    4. 媒体网站从胜出的 DSP 拿到广告创意并展示

      2.3 和 2.4 可以合并，即 DSP 将出价和广告创意一起发送给 ADX，好处是减少一次服务次往返，延迟低，缺点是 ADX 可以获得 DSP 某个广告商的相关受众，存在信息泄漏风险，不太符合 ADX 中立市场的地位

  • 实时竞价的交易方式，带来了以下问题：
  1. 每次展示都有 ADX 与多个 DSP 服务器的参与，这使得服务器与带宽成本大大增加
  2. 在询价过程中，ADX 要等待一个约定好的时间片（一般为 100 ms），这使得用户看到的广告延迟增加，对 CTR 有负面影响。（移动应用客户端可以缓存广告创意）
  3. DSP 可以以极低的出价参与竞价，低成本得到媒体的用户行为数据，存在信息泄漏风险
  • RTB 的接口有两个对接方
    • RTBD：在 ADX 方实现的部分
    • RTBS：在 DSP 方实现的部分
  • 实时竞价中的「实时」，指的是需求方在每次广告展示时参与竞价，而非供给方实时比价的过程

2 其他程序化交易方式

2.1 优选（PD）

  • 优选（Preferred Deal，PD）比实时竞价早，可以看成是只有一个需求方的程序化交易
  • 优选允许单个需求方既可以按照自己意愿挑选流量，又可以避免复杂的竞价过程
  1. 用户访问媒体页面
  2. 媒体页面通过 JS 代码或 SDK 向供给方广告投放机发起广告请求（可以是媒体自己的广告投放机，也可以是其他供给方产品的）
  3. 供给方广告投放机向需求方广告投放机发起请求，询问是否需要此次展示机会
  4. 若需求方不需要此次展示，则通知供给方广告投放机，从自有广告库选择合适的创意返回
  5. 若需求方需要此次展示，则返回相应的广告创意
  • 优选一般按 CPM 结算，由于没有竞价，又有选择流量的便利，价格一般比市场价格高
  • 优选的一个缺点是，决策过程可能存在多次投放机往返

2.2 私有市场（PMP）

  • 有时媒体为了保证广告质量，将拍卖限制在一些被邀请的需求方的小范围内，这种程序化交易称为私有市场
  • 私有市场的在线交易过程与公开的实时竞价一致
  • 私有市场兼顾了优选和实时竞价的好处
    • 邀请制而非公开的交易方式，有利于控制广告质量，确保媒体的价值不被伤害
    • 被邀请的各方仍保留了竞价的关系，有利于提升媒体的变现能力
    • 私有市场使更多的优质媒体加入程序化交易市场，促进了品牌广告投放的程序化进程

2.3 程序化直投（PDB）

  • 属于直接购买与实时竞价之间过渡的一种交易模式
  • 交易本身仍以定价、保量或半保量的方式完成，但需求方可自行对采买的库存做广告投放决策，有时可进行一定的流量选择
  • PDB 适用于以下场景
  1. 跨媒体频次控制：在合约广告模式中，DSP 代表广告主进行广告投放，可以进行跨媒体的频控
  2. 多个子产品流量分配
  3. 一定比例的还量

2.4 广告交易方式谱系

  • 优先销售：将广告投放的关键要求约定下来的交易方式，其中人的因素起了相当大的作用，这种售卖一般能满足品牌性需求，溢价能力较强，属于高质量媒体优先考虑的售卖方式（按时间段和广告位独占式售卖，展示量合约售卖）。技术主要集中在供给方（在线分配、受众定向）
  • 网络优化（network optimization）：在竞价广告网络中，供给方同时使用多个广告联盟，按照一些规则或模型对不同的流量分割选择不同的 ADN

3 广告交易平台

  • 负责将媒体流量以拍卖的方式售卖给 DSP，可以类比证券市场中的交易所
  • ADX 的产品策略较为简单，不需要保存广告库，不需要广告检索流程，排序过程也简单
  • ADX 需要注意的产品策略，主要是如何解决给多个 DSP 发广告请求带来的，带宽和机器成本的上升。这一问题称为询价优化（具体技术方案在 15 章介绍）
  • 一般按 CPM 结算，原因在于第一方数据在程序化交易中的广泛使用，DSP 对于人群在特定广告上产生的效果，往往能较为准确的估计
  • 产品案例：DoubleClick ADX

4 需求方平台

  • 程序化市场中的需求方产品是 DSP，其核心特征有两个：
    • 支持 RTB、优选等程序化的流量购买方式
    • 支持需求方定制化用户划分（定制化用户标签）
  • 所谓定制化用户划分，即形如 tag(a, u) 的定向方式，两个例子：
    • 电商网站的「老客户」
    • 银行信用卡的「潜在客户群」

4.1 需求方平台产品策略

两个关键的产品策略：

  1. 出价
  2. 受众定向的方式。DSP 中，主要的受众定向方式是，以第一方数据为核心，结合第二方或第三方数据的定制化标签，即 t(a, u) 的定向方式。其中，重定向和新客推荐（look-alike）比较重要。

4.2 出价策略

  • 在 DSP 中，准确地估计 eCPM 非常关键
  • 如果没有预算的限制，按照 eCPM 出价就是最优策略
  • ADX 的成交价在一天内随时间会剧烈变化
  • 假设我们的 eCPM 是高于市场水平的固定值，可以比较两种出价策略：
    • 策略 A：对所有询价按 eCPM 出价，这样可以获得所有流量，直至当日预算消耗完
    • 策略 B：选择一些市价较低的流量出价，获得这些流量，直至当日预算消耗完 > 显然，策略 B 的成本低于策略 A，当然也能获得更高的利润
  • 上述 DSP 的出价策略可以定性描述为：
    1. 通过历史的观察和预测，得到市价的曲线
    2. 将一天的预算分配到那些市价较低的流量上 > 实际情况更复杂，因为 eCPM 也会随时间变化（例如游戏广告在休闲时间比上班时间的 eCPM 高），因此我们希望获得的是 eCPM/市价 较高的流量。故 DSP 的出价策略基于两条曲线，即 eCPM 和市价随时间变化的曲线。（除了时间，也有其他影响策略的因素，比如性别）

4.3 出价和定价过程

  • DSP 同时存在面向广告主的定价过程和面向 ADX 的出价过程
  • DSP 向广告主定价、ADX 向 DSP 定价时，都采取了第二高价，DSP 获得了 b - b’ 的期望收益

4.4 重定向

  • 重定向时在线广告中最早产生，也是最广泛使用的一种定制化标签
  • 有以下两种目的：
    • 用于品牌广告。当用户已经选择过某品牌的服务或产品后，如果在质量比较高的媒体上看到该品牌的广告，这个用户会进一步肯定自己决策的正确性，从而对该品牌的认知度也大大加强。这种用途下，应当以宣传品牌，而不是具体产品为主要诉求。
    • 用于效果广告。当用户曾经考虑过某种产品，但没有完成最终转化，通过在线广告将这个用户找回，点击率和转化率都会明显高于平均水平。如果用户已经选择了该产品，那么可以利用推荐技术向这个用户推送相关的产品广告。
  • 根据信息来源和使用信息的精细程度，重定向有以下类型

  1. 召回重定向与个性化重定向

    • 网站重定向（site retargeting）：将一段时间内到达广告主网站或应用的用户作为重定向集合，这样的重定向流量的 eCPM 一般比无定向流量高出一个数量级

    • 个性化重定向（personalized retargeting）：按照品类和购买阶段等因素进行创意上的深度个性化。可以在两个方面做深入挖掘：

      1. 对于处于不同购买阶段的用户，采用合适的创意推动他尽快完成转化行为，购买阶段包括浏览、搜索、加入购物车等
      2. 对于已经有过购买记录的用户，使用推荐技术向其展示相关的商品，以提高二次购买率

    • 从广告产品的视角看，称为个性化重定向；从推荐产品的视角看，可以认为是站外推荐（off-site recommendation），即，将原来广告主网站上的推荐模块搬到了站外

    • 个性化重定向与传统的受众定向方式有较大差别：

      1. 动态创意。由于广告主商品数量巨大，无法为所有商品组合预先准备好创意，动态创意是个性化重定向最重要的支持技术。
      2. 推荐引擎。个性化重定向可以看作是站外推荐，与站内推荐有以下区别：1. 站内推荐主要根据上下文信息进行，站外推荐主要是根据用户信息进行 2. 站内推荐一般发生在某个特定的购买环节，因此不需要动态调整创意，而站外推荐需要
      3. 广告主商品库存实时接口。个性化重定向服务需要提供准实时的商品库接口，让广告主可以及时地将库存与价格信息同步过来

  2. 搜索重定向（search retargeting）

  • 将搜索过与广告主直接相关的关键词的用户群作为重定向集合
  • 也可以获得很精准的用户群，其绝对量也要高于网站重定向
  • 但对于非主要搜索引擎提供商来说，依靠搜索重定向能覆盖的人群比例未必会高于网站重定向
  • 确定合适的搜索重定向词表，有如下 3 种思路：
    1. 人工根据经验确定词表
    2. 如果该广告主同时也在搜索引擎投放，可以直接采用对应投放的关键词表
    3. 基于广告主流量统计方法，即统计广告主网站流量中从搜索引擎来的流量，将其中频度较高的关键词作为搜索重定向词表
  • 第三种策略有 2 个优势：1. 根据实际流量数据得到，效果往往更好 2. 通过在广告主网站布置代码等手段得到第一方数据后，整个过程可以自动化，简便易行

4.5 新客推荐

  • 广告主根据自己的第一方数据提供一部分种子用户，再由拥有更丰富数据的第二方数据的广告平台分析这些用户网络行为的特征，找到具有相似特征的拓展人群
  • 考虑的是没有广告主站内行为情况下的推荐，是原有狭义推荐问题的扩展
  • 新客推荐要在一定数据支持下才有可能产生价值，并且由于它涉及第一方数据和第三方数据的获取和加工，在技术上具有挑战性

4.6 产品案例

Criteo, InviteMedia

5 供给方平台

  • 媒体的变现平台可以采用如下产品逻辑：
  1. 首先检查优先销售的订单有无需求（CPT 和 CPM 合约），若有，则按优先级和在线分配的方案完成投放
  2. 如果没有优先销售合约，则进入竞价流程：从自运营广告主库中找出 eCPM 较高的，并估算可供调用的若干广告网络的 eCPM，在这两者间找到较高的，以此作为 MRP 通过 RTB 接口，向接入的各 DSP 实时询价。
  • 可以看出，广告请求是被分配到自运营广告库，还是对接其他广告网络，或是以 RTB 方式对接 DSP，是根据其收益动态计算决定的，该方案称为动态分配（dynamic allocation）。对应的产品形态称为供给方平台（supply side platform，SSP）

5.1 供给方平台产品策略

1. 聚合（mediation）：集成多个 ADN 的 JS 代码或 SDK，在线动态决定向谁请求广告。选择 ADN 的逻辑，即为根据某次展示机会的 (u, c)，对候选的 ADN 估计 eCPM。由于此时没有具体广告，只能粗略估计，实际中考虑的因素主要有：1. ADN 的分成模式和比例 2. ADN 的广告返回率 3. 如果某 ADN 主要投放某种行业类型的广告，那么 (u, c) 中的人群和环境标签也有帮助 2. 广告投放（ad serving）：估计媒体自行销售的广告的 eCPM，像是帮助媒体实现了一个私有的 ADN 3. 市场（marketplace）：SSP 以 RTB 方式向 DSP 请求广告，这形成了一个市场。SSP 里的市场与 ADX 没有本质区别，尤其当 DSP 越来越多，ADN 越来越少时，SSP 与 ADX 呈现出越来越强的同质化趋势 - 有时 SSP 向选出的 ADN 投放请求后，该 ADN 不一定能填充这次展示机会，那么 SSP 就要跳转向排名第二的需求方，若还是 ADN，则再次请求广告，依此类推，该模式称为瀑布模式。 - 瀑布模式中的多次请求使得总体请求广告的延迟大大增加，比较低效，Header Bidding 模式在一定程度可以解决此问题。

5.2 Header Bidding

  • Header Bidding 时 Bidder 和媒体建立直连，绕过 ADX 的一种方式。Bidder 有机会在实时竞价开始前就直接向媒体报价，媒体根据报价高低决定中标者。这里的 Bidder 指所有可以进行报价的服务，可以是 DSP，也可以是 ADX 或其他广告参与者。
  • 直接报价的过程，发生在客户端，即浏览器或应用中。
  • Header Bidding 的广告决策过程如下：
  1. 用户访问媒体页面，向媒体服务器发起 http 请求
  2. 媒体服务器将实现 Header Bidding 功能的脚本 hb.js 放在 HTML 的 head 标签中，该 HTML 作为 HTTP 响应发给用户浏览器
  3. 用户浏览器在解析 HTML 时，将媒体网站配置好的 hb.js 下载到本地。在 hb.js 的控制下，用户浏览器向媒体网站约定好的 Bidder 发起本次曝光机会的竞价请求，Bidder 将报价返回给用户浏览器
  4. 在 hb.js 控制下，用户浏览器将各家 Bidder 报价信息回传给媒体网站
  5. 媒体服务器同时向 ADX 或 SSP 发送广告请求
  6. ADX 或 SSP 发起 RTB 过程并获得广告候选
  7. 媒体服务器将 Header Bidding 出价结果和 RTB 出价结果放在一起进行排序，出价最高者赢得本次广告展示机会，用户浏览器请求胜出方加载广告
  • 客户端询价的方式带来了比较大的广告时延，这使得实际的曝光减少了，也影响了用户体验
  • Header Bidding 的成功更是一种商业博弈的结果，而非技术上的进步
